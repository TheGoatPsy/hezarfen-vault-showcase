# MOC — Map of Content Pattern

A Map of Content (MOC) is a single markdown file that acts as the navigation hub for a vault. It is not an index of every file; it is a curated map of the domains a researcher actually moves between, with a handful of entry points per domain.

This file shows the pattern using placeholder domains. A real vault might slot in clinical practice, academic research, software projects, client work, personal finance, and so on. The pattern is the same regardless of domain.

---

## Structure

A working MOC has three layers:

1. **Top-level domains** — 5–10 buckets that describe what the person actually does.
2. **Active entry points per domain** — links to the 2–5 notes that a researcher currently opens most often in that domain. These rotate as projects start and finish.
3. **Archive pointer per domain** — a single link to a `Archive/` or `Completed/` folder where inactive material moves once a project closes.

The MOC is edited manually and by hand. It is deliberately short (under 200 lines) so it can be scanned in a few seconds. Automation does not generate this file — the researcher chooses what is important.

---

## Template

```markdown
# Map of Content — [Researcher Name]

Last updated: YYYY-MM-DD

## Domain A — [e.g. Clinical Practice]

Active:
- [[active-case-notes]]
- [[current-supervision-log]]
- [[license-renewal-tracker]]

Reference:
- [[practice-standards]]
- [[referral-network]]

Archive: [[Domain-A-Archive]]

## Domain B — [e.g. Academic Research]

Active:
- [[manuscript-in-revision]]
- [[grant-deadline-tracker]]
- [[reading-queue-2026-Q1]]

Reference:
- [[methodology-library]]
- [[coauthor-contacts]]

Archive: [[Domain-B-Archive]]

## Domain C — [e.g. Software Projects]

Active:
- [[current-sprint]]
- [[bug-triage]]

Archive: [[Domain-C-Archive]]

## Cross-cutting

- [[monthly-finance-ledger]]
- [[calendar-2026]]
- [[weekly-review-template]]

## Session Anchors

- [[CLAUDE.md]] — canonical rules
- [[MEMORY.md]] — memory index
- [[current-daily-log]] — today's session
```

---

## Why this shape

- **Top-level domains stay small.** If a researcher works in five domains, the MOC has five sections. Adding a sixth forces an explicit decision about whether the work belongs in the vault at all.
- **Active entry points surface current work.** The MOC is not a complete inventory; files that are genuinely active are linked, others are not. When a project closes, its entry points move to the archive pointer.
- **Archives are referenced, not inlined.** Each domain gets a single archive link. Inactive material is findable but does not clutter the map.
- **Cross-cutting notes live at the bottom.** Finance, calendar, templates — things that do not belong to any single domain — get their own section.
- **Session anchors sit at the very end.** These are the notes that the AI agent or the researcher opens every session regardless of which domain they are working in.

---

## What the MOC is not

- It is not a tag cloud.
- It is not a search index.
- It is not generated by a plugin or script.
- It does not list every file.

If the MOC starts to look like any of those things, it has stopped being a map and become a database. Trim it back.

---

## Maintenance

A reasonable cadence is:

- **Weekly:** prune active entry points that have not been opened in 14 days; move them to archive or delete the link.
- **Per project close:** move the project's entry points into its domain archive pointer.
- **Per new project:** add an active entry point at the top of the relevant domain.
- **Quarterly:** reconsider whether the top-level domain list still matches the researcher's actual work; adjust if drifted.

The MOC is the table of contents for a life's work-in-progress. It earns its keep only if it is kept current.
